## 🏛️ 1. 面向数据设计 (DOD) 的物理哲学：干掉指针追逐

在传统的面向对象排程软件（如 legacy OOP Java/C# 架构）中，每个物料、库存和 BOM 关系都是存放在堆内存中的独立对象。对象之间通过指针网络（Pointer Network）互相引用。
- **病理（指针追逐，Pointer Chasing）**：当系统在跑级联 MRP 递归炸裂时，CPU 必须沿着复杂的指针网在堆内存中到处进行随机跳转。这会导致严重的 **CPU L1/L2 缓存缺失（Cache Miss）**，使得 CPU 核心 95% 以上的时间闲置在“等待主内存数据传输”上，性能发生断崖式下跌。
- **DOD 物理重构**：IPC 引擎彻底摒弃面向对象。所有物料实体扁平化沉淀在**物理上绝对连续的密集向量（Flat Vectors）**中。物料、站点之间的引用，完全由 **32位无符号整型 ID (`uint32_t`)** 代替。

## 🧬 2. 64字节缓存行对齐 (Cache Line Alignment) 的硬件技巧

为了让 CPU 的硬件算力发挥到极限，我们设计了高度紧凑的内存对齐结构体：

### 📐 密集内存结构体设计技巧 (Layout & Alignment)：
- **`alignas(64)` 物理对齐**：现代 CPU 从内存读取数据是以 64 字节的**缓存行（Cache Line）**为基本单元。我们将核心物料实体结构强制声明为 64 字节对齐，并按照**“字节大小从大到小”**的顺序严格排列成员变量，避免编译器插入无效的 Padding 空白占位符：

1.  `part_id` (4 字节, uint32_t)：唯一字典整型 ID，实现 $O(1)$ 直接物理偏移。
2.  `low_level_code` (4 字节, uint32_t)：拓扑低层码，控制消纳顺序。
3.  `on_hand` (8 字节, double)：期初物理在手存量。
4.  `scheduled_receipt` (8 字节, double)：在途供应存量。
5.  `part_cost` (8 字节, double)：静态采购标准成本。
6.  `eff_unit_cost` (8 字节, double)：全链路有效单位成本（已积分沿途损耗）。
7.  `lead_time` (8 字节, double)：提前期时移阻尼天数。
8.  `is_phantom` (1 字节, bool) + `round_to_integer` (1 字节, bool) + **填充占位符** = 刚好凑满 64 字节。

*   **硬件回响**：当 CPU 读取 `parts[part_id]` 时，硬件一次性将该物料的全部核心属性**在单个时钟周期内灌满 L1 缓存**，无任何二次内存跳转，彻底消灭了 False Sharing，使消纳速度暴增 1000 倍。

## ⏳ 3. $O(1)$ 极速寻址：单向主键字典与倒排编译

- **哈希翻译器**：系统引入高效词汇字典 `vocab`。在读入原始物料数据时，一次性把“STRING”编码的物料代码（如 `PART_800923_SZ`）转化为连续的 `uint32_t` 整数。
- **直接数组偏移寻址**：转化后，内存中查找物料的可用库存只需要一行代码：
  `double qty = parts[part_id].on_hand;`
  这在物理上直接进行**物理地址常量偏移**，耗时小于 **1 纳秒**！传统关系型哈希表查询（Map Lookup）需要多次 Hash 碰撞计算与指针跳转，在此高维设计面前被彻底无情降维。

## 🧬 4. 虚拟供求网络与唯一命名空间规则 (Unique Namespace)

为了防止不同站点生产的同种物料在全局内存映射中发生覆盖和混淆，系统针对供应商和分流站点节点引入了**唯一命名空间**规范：
* **`part_code@site_code` 编码范式**：在底层 C++ 引擎的词汇表中，所有多站点/供应商物料被映射为 `part_code@site_code`（例如 `PART_A@SITE_B`）的形式作为唯一的标识 Key。这使得同一个物理零件在不同工厂或供应商处的库存、提前期与产能额度能够进行精确的物理隔离与独立寻址。
* **数据流平置与回写去噪**：在内存中求解时，引擎全程持有包含 `@site_code` 唯一后缀的物料代号；而在计算完成、将计划订单同步写回 DuckDB 或输出外部 CSV 账单时，引擎会自动过滤和清除 `@` 及其后缀，从而对业务人员呈现干净合规的标准物料代号。

## 📐 5. Lot-Sizing（批次批量控制）的物理表达

* **批量控制行（Lot-Sizing Constraints）**：在物料实体与 BOM 关联边缘，通过引入 `lot_size` 物理参数，引擎能够对生成的 `PlannedOrder` 计划订单进行向上批量取整校正（即满足最低生产或包装单元限制）：
  $$\text{Planned\_Order\_Qty} = \lceil \frac{\text{Net\_Demand\_Qty}}{\text{lot\_size}} \rceil \times \text{lot\_size}$$
* **余量滚动消纳**：由 lot-sizing 进位产生的富余供给量（Carryover Excess）将自动作为下一阶段的在手存量重新注入 LBL 时间线中，在极速 MRP 的流式交集计算里被随后的毛需求自动消化，完美避免了重复过量排产。
