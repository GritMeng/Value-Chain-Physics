# 专利技术交底书

## 一、 发明创造的名称
一种基于嵌入式列式内存数据库物理沙箱的多版本 What-If 协同规划系统及方法

---

## 二、 本发明的技术领域
本发明属于企业级高级计划与排程（APS）、多维供应链控制塔、What-If 计划模拟沙箱、嵌入式分析型数据库（OLAP）以及多版本并发控制（MVCC）技术领域。

---

## 三、 背景技术与现有技术的缺陷
在现代离散制造及复杂供应链管理中，**What-If 模拟分析（场景规划 / Scenario Planning）** 是计划控制塔决策的核心诉求。当供应链面临突发的外部扰动（如关键供应商原料交付延误、生产线突发故障、客户紧急插单等）时，计划人员需要在不影响在线生产环境（Baseline）的前提下，快速克隆出一个或多个平行的虚拟计划场景，在沙箱中调整参数（如加急空运、安排设备加班、升级芯片规格），并运行重跑计划引擎，以比对不同决策对全局 KPI（如订单交付率、呆滞库存成本、 Consensus 营收）的物理影响。

然而，现有的高级计划排程（APS）与控制塔系统在处理多版本 What-If 协同模拟时，面临极其严重的架构与性能瓶颈：
1.  **图节点多版本并发控制（MVCC）的内存灾难。**
    诸如 o9 Solutions 等主流规划系统通常采用统一的 Enterprise Knowledge Graph（企业知识图谱）架构。为了实现 What-If，它们在物理共享内存中对图节点的边和属性进行逻辑上的“多版本打标（Versioning / Layering）”。这种设计在面对包含数百万订单、物料和 BOM 关系的超大规模供应链网络时，会导致内存指针级联分裂、堆内存空间呈指数级膨胀，并引发高频的垃圾回收（GC）停顿。同时，复杂的版本合并（Merge）和冲突检测算法逻辑极度臃肿，开发和维护成本高昂。
2.  **高频计算引擎与分析型数据库的对账割裂。**
    高性能计划引擎（通常为裸金属 C++ 开发）为了榨干 CPU 缓存，采用连续扁平内存（DOD/SoA）进行拓扑计算。而计划人员和前端控制塔则习惯使用关系型 SQL 语句对计算结果进行复杂的切片、多维分析（OLAP）与明细穿透下钻（Drill-down）。在传统架构中，C++ 引擎的内存计算结果必须跨网络导出至外部的大型关系型或列式数据库中，同步延迟长达数分钟，导致“What-If 实时刻度对账”根本无法实现，破坏了控制塔的毫秒级交互体验。
3.  **多代数据定义语言（DDL）平滑迁移机制的缺失。**
    在系统升级、场景合并或表结构变更时，由于传统规划系统的大型数据库表依赖大量的索引（Indexes）与外键约束，执行表结构重命名（RENAME TO）或字段重构极易发生锁表，甚至因为索引依赖冲突报错挂起，无法在不中断在线计划服务的前提下完成库表自适应演进。

---

## 四、 本发明要解决的技术问题
本发明的目的在于提供一种基于嵌入式列式内存数据库物理沙箱的多版本 What-If 协同规划系统及方法，用以解决传统规划系统在 What-If 模拟中图版本内存开销巨大、高频计算与 SQL 分析对账割裂、以及场景迁移重命名冲突的技术痛点，实现秒级场景物理隔离克隆、零拷贝直接 SQL 查账与自适应无冲突 DDL 平滑升级。

---

## 五、 本发明的技术方案（核心创新点及具体实现）
为了实现上述目的，本发明提供的技术交底方案包括以下五个层面的核心实现：

### 1. 嵌入式列式数据库物理沙箱隔离机制
系统放弃在单个大图数据库上进行繁琐的版本打标，转而采用**进程内共生嵌入式列式分析数据库（DuckDB）物理文件克隆机制**。
系统将生产底座（Baseline）的主数据、需求及计划结果完全存储于一个单文件列式数据库 `ipc.db` 中。当计划人员触发创建新场景 $S_i$ 时，API 服务通过操作系统底层快速文件流拷贝，在零阻塞主服务的前提下将 `ipc.db` 瞬间克隆为 `sandbox_Si.db`：
```python
shutil.copyfile("ipc.db", "sandbox_Si.db")
```
在主数据库与沙箱数据库中，均维护一张场景注册元数据表 `ipc_collab_scenario`。克隆物理文件 `sandbox_Si.db` 时，同步复制包括多工厂替代路由（`ipc_alternative_routing`）、刚性配额限制（`ipc_allotment_quota`）、采购比例组均衡规则（`ipc_purchasing_ratio`）及不完全替代限额（`ipc_substitution_rules`）在内的完整关系模型。克隆完成后，API 服务将沙箱状态标记为 `ACTIVE` 并更新连接指针，实现沙箱的完全物理隔离，从而在进程和物理文件层面消除了任何多版本数据踩踏风险。

### 2. 裸金属 C++ DOD 计算层与嵌入式列式数据库的零拷贝同步管道
C++ 计划求解器引擎与嵌入式列式数据库在同一个操作系统进程内直接通过内存总线对接。
当求解器触发计算时，计算引擎从 `sandbox_Si.db` 提取数据，并通过 PartSite 词汇表（PartVocab）将字符串 Code 转换为扁平 `uint32_t` 类型的数值 ID，常驻于扁平 SoA 连续物理内存中进行 MRP 与有限产能调度。
计算完毕后，C++ 引擎使用 DuckDB 的 `Appender` 并行灌库管道，将 `ipc_planned_order_ledger`（计划订单）、`ipc_planned_supply_assignment`（供给分配账本）、`ipc_financial_ledger`（财务账本）等计算明细直接写入沙箱数据库，写入速率达每秒千万级：
算法：基于内存总线与流式对账的零拷贝追加更新流程
输入：C++ 计算核心产出的计划订单集合 planned_orders，动态替代跟踪集合 substitution_traces，数据库连接句柄

对于计划订单表（`ipc_planned_order_ledger`），按行直接在共享内存中以追加方式写入以下字段：
1) 订单唯一编码 (`order_code`)
2) 物料编码 (`part_code`)
3) 计划数量 (`order_qty`)
4) 计划开工日 (`start_day`)
5) 计划完工日 (`finish_day`)

对于物料动态替代跟踪表（`ipc_substitution_trace`），以追加方式写入以下字段：
1) 原始物料编码 (`original_part_code`)
2) 替代物料编码 (`substitute_part_code`)
3) 原始需求数量 (`original_qty`)
4) 实际分配的替代数量 (`substitute_qty`)
5) 动态替代比例因子 (`substitution_ratio`, 即 β)
6) 动态替代上限约束 (`substitution_cap`, 即 Cap)

关闭追加句柄，完成零拷贝追加更新，数据在嵌入式分析数据库中即时生效。
这一管道直接在物理内存中以扁平二进制段形式追加写入，彻底摒弃了繁琐的 `INSERT` 关系型 SQL 解析过程，消除了 SQL 编译与网络传输开销，实现了计算引擎与查账分析层的零延迟同步。

### 3. 多代 DDL 自适应平滑重命名升级机制
在计算引擎点火运行或场景重整阶段，系统会自动触发自适应表结构迁移：
1.  **强行清空挂起索引**：首先执行 `DROP INDEX IF EXISTS`，完全移除诸如 `idx_po_part` 等作用于目标表的旧物理索引，解除表重命名的外键与物理依赖。
2.  **多代平滑重命名升级**：扫描历史遗留表名（如 `part_site`、`independent_demand`、`coproduct_schedule` 等），执行 `ALTER TABLE ... RENAME TO ...` 将其统一无缝迁移至全新的规范化命名空间（如 `ipc_onhand`、`ipc_independent_demand` 等）。
3.  **拼写及字段自适应纠错**：检测字段拼写变异（例如将 `deminsion` 或 `demision_grp` 自动规整为 `dimension_grp`），动态创建标准表，复制数据后 `DROP` 旧表，确保计算引擎的输入字段 100% 正确。

### 4. 控制塔双轨制 Scenario 交互与合并路由
系统对外暴露标准化 RESTful API，控制塔前端可视化界面通过动态参数 `?scenario=Si` 与沙箱交互。
*   **读取路径**：前端调用 `/api/table` 或 `/api/kpis` 时，API 服务根据当前激活的 `ACTIVE_SCENARIO` 路由将 DuckDB 连接定位到对应的 `sandbox_Si.db`，直接使用标准 SQL 语句执行海量明细数据的毫秒级 OLAP 穿透下钻。
*   **合并路径**：当计划人员确认沙箱场景 $S_i$ 的 What-If 结果优于基线并决定发布合并时，API 服务首先将当前活动连接指针强制重置指向 `ipc.db`（Baseline），随后通过操作系统文件锁，以 `sandbox_Si.db` 强制覆盖 `ipc.db`，最后删除沙箱文件，完成“基线升级”，实现了大盘计划的安全、零冲突更替。

## 六、 本发明的技术效果及创新点
1.  **物理隔离带来绝对的并发安全**：
    以操作系统物理文件克隆代替逻辑图版本打标，彻底避免了多版本带来的内存泄露与 GC 踩踏问题，沙箱创建耗时仅为毫秒级。
2.  **零延迟 SQL 列式分析对账**：
    C++ 内存计算通过无锁 Appender 管道与同进程 DuckDB 内存对账，既保证了求解器的极速裸金属 DOD 计算，又赋予了控制塔前端毫秒级 SQL 穿透与 KP I实时分析的超凡体验。
3.  **安全无冲突更替与自适应演进**：
    设计了清空索引重命名与物理文件合并双路由，以及 DDL 自适应重定义拼写纠正，确保了供应链数据库在长期运行中具备极强的抗熵平滑升级能力。

## 七、 实施例（数值演算）

本实施例以一个实际的“What-If 插单决策”场景为例，演示本发明的数据库沙箱创建、计算灌库、SQL 对账与最终合并发布的全过程。

### 1. 基线（Baseline）数据状态
主数据库 `ipc.db` 中有 300 万笔需求订单，其中关于核心成品 `FG_Mate70` 的需求总量为 100,000 颗，期末财务 Consensus 总营收为 15.6 亿元。

### 2. 沙箱创建与隔离
计划员发现收到了一笔大客户的 VVIP 紧急插单 `Order_VVIP`（Day 5 需要 5000 颗 `FG_Mate70`）。为评估插单风险，计划员在控制塔前端触发沙箱创建：
*   **动作**：API 服务调用沙箱克隆。
*   **状态变化**：
    1. 生成物理文件 `sandbox_vvip_insert.db`（耗时 12 毫秒）。
    2. 主数据库与沙箱数据库的 `ipc_collab_scenario` 表同步写入注册项：
       `('vvip_insert', 'VVIP插单评估沙箱', 'User', now(), 'ACTIVE')`
    3. `ACTIVE_SCENARIO` 动态指向 `vvip_insert`，`DB_PATH` 重定向为 `sandbox_vvip_insert.db`。

### 3. 沙箱参数调整与 C++ 求解器计算
1.  计划员在沙箱中写入插单需求，并在 `ipc_dispatch_ledger` 表中临时将 Day 5 设备的最大产能上限（`max_capacity_batches`）扩容至 120%（批准设备加班）。
2.  调用 `/api/run?scenario=vvip_insert`。API 服务以命令行参数形式启动 C++ 引擎：
    `main_mem3.exe --db sandbox_vvip_insert.db`
3.  C++ 求解器在连续物理内存中完成 LBL-MRP 级联消纳与 DBD 有限产能时空派程，计算耗时 240 毫秒。计算完毕后，C++ 引擎通过 `duckdb::Appender` 流式向 `sandbox_vvip_insert.db` 灌入更新后的 3,005,000 条计划订单和产能分派账本，灌库耗时 80 毫秒。

### 4. 控制塔 SQL 对账比对
计划员在控制塔前端切换基线与沙箱，进行 KPI 比对。API 服务对两个物理 DB 执行 SQL 查询：
*   **对基线（ipc.db）执行查询**：
    `SELECT sum(total_revenue) FROM ipc_financial_ledger` ➔ 返回 `1,560,000,000` 元。
*   **对沙箱（sandbox_vvip_insert.db）执行查询**：
    `SELECT sum(total_revenue) FROM ipc_financial_ledger` ➔ 返回 `1,595,000,000` 元。
*   **对沙箱产能分派明细执行下钻查询**：
    `SELECT allocated_capacity FROM ipc_dispatch_ledger WHERE scheduled_day = 5` ➔ 返回 `120.0%` (确认加班负荷合理)。

对账结果显示，该 VVIP 插单决策将使企业净营收增加 3500 万元，且 Day 5 的加班负荷在可控范围内。

### 5. 沙箱合并与发布
计划员点击“发布合并（Publish Merge）”：
*   **动作**：
    1. API 服务锁闭写通道，将 `ACTIVE_SCENARIO` 强制切回 `baseline`。
    2. 释放对 `ipc.db` 与 `sandbox_vvip_insert.db` 的活动句柄连接。
    3. 执行物理文件替换：`shutil.copyfile("sandbox_vvip_insert.db", "ipc.db")`（耗时 15 毫秒）。
    4. 执行文件删除：`os.remove("sandbox_vvip_insert.db")`。
*   **结果**：大盘数据安全、零冲突升级。控制塔大盘正式更新 Baseline 营收为 15.95 亿元，What-If 决策流式闭环完成。

## 八、 与先前及后续专利的关联关系

本专利作为“基于内嵌列式内存数据库物理隔离克隆的 What-If 多版本协同沙箱”，为整个 IPC 系统的算法和决策提供了高安全、低开销的事务与文件级隔离机制，它与本系统的其他专利形成了紧密的底层级联：
1. **与专利 1（一种基于解耦点刚性配额与优先级排程的双层规划系统及方法）的关联**：
   专利 1 的双层规划在进行宏观 ITP 大盘冲销与微观 IOP 优先级交期排产时，其读写操作均重定向至本专利所克隆出的物理沙箱文件 `sandbox_Si.db` 中。这保证了用户并发进行 What-If 模拟时，计划大盘的配额 Allotment 逻辑不会发生数据踩踏。
2. **与专利 2（一种基于联副产品分级分流消纳的配产控制系统及方法）和专利 3（一种基于时域并行前缀和分治的同节点ATP供需平衡消纳系统及方法）的关联**：
   专利 2 的凸集投影迭代算法与专利 3 的时域二叉段树消纳算法，在进行大规模运算时需要获取物理在库与路线参数。它们通过本专利的零拷贝内存 Appender 直通管道，在同进程内将计算出的投产批数、降级代用量、以及时序 OnHand 水位高速灌入当前沙箱，避免了昂贵的 SQL 解析和网络传输。
3. **与专利 8（一种基于物料依赖 LLC 拓扑分级的大大规模 MRP 并行消纳调度系统及方法）的关联**：
   专利 8 在对数百万级 SKU 执行层内 OpenMP 并行 MRP Netting 展开前，系统会调用本专利瞬间克隆出物理沙箱。专利 8 的所有多线程并发扣减直接在沙箱的物理内存段上执行，消除了传统多核 MRP 在主基线数据库上的行级锁死锁冲突。
4. **与专利 9（一种基于原子级乐观时间戳锁的跨区域/渠道虚拟安全库存池化共享系统及方法）的关联**：
   当多线程在沙箱内并发执行库存扣减时，它们使用专利 9 的乐观时间戳版本 CAS 控制。当 CAS 替换成功并确认扣减后，分配流通过本专利的 Appender 流式追加写入沙箱数据库，保证了虚拟池化分配在沙箱中的审计一致性。
5. **与专利 10（一种基于线程局部零堆分配回滚栈与时空网格的有限能力排产系统及方法）的关联**：
   在沙箱内运行 IOP 有限能力排程时，排产线程使用专利 10 提供的线程局部零堆分配栈记录尝试性扣减。如果 CTP 排产判定失败，回退栈在沙箱对应的内存映射段中原位反向加回，整个过程不向本专利的物理沙箱表中写入任何脏数据。
6. **与专利 11（一种基于数据模型同构双螺旋与自反馈控制的抗熵计划调节系统及方法）的关联**：
   当物理探针反馈车间异常导致残差超标时，专利 11 自动调用本专利，瞬间克隆出一个纠偏 What-If 沙箱，在隔离环境下运行自反馈纠偏计算。寻找到了能够消纳偏差的 CDM 参数后，通过本专利的物理覆盖机制强制合并升级 Baseline，阻断了熵增在主系统的扩散。
