## 🔎 CASE 1：双阶段配额隔离消纳与共享剩余调剂 (Allocation Netting & Residual Sharing)

### (1) 产品说明书（业务规则）
在高频变动的销售执行期（IOP），为防止低优先级订单抢占 VVIP 订单的战略备料，同时避免因过度硬隔离导致重复采购（例如：A区域缺料而B区域配额多余却不能共享，导致系统又下达采购单），IPC 引入**双阶段配额消纳与剩余调剂机制**：
*   **空间维度隔离**：订单按 `[Region 区域 + ProductGroup 产品组 + CustomerGroup 客户组]` 空间属性匹配配额（Allocation）。
*   **第一阶段（硬隔离消纳）**：订单仅能消费自身组内、在**双周滚动窗口**（当前周 $W\_0$ + 下周 $W\_1$）内的 Allocation。
*   **第二阶段（共享剩余调剂）**：若订单在第一阶段仍有缺口，在拉动新供应之前，系统收集全网其他组在 $W\_0$ 和 $W\_1$ 内未消耗完的**剩余配额（Leftover）**进行调配满足。
*   **净需求触发**：只有当第一阶段专有配额和第二阶段共享剩余均被耗尽后，最终的缺口（Delta）才作为净需求，拉动并产生新的计划订单。

### (2) 设计说明书（数据结构与计算步骤）
#### 核心数据结构：
*   `ipc_allocation_ledger` (配额账本表)
*   `ipc_independent_demand` (独立需求表)

#### 时序计算步骤（以 Day 3 订单为例）：
1.  **确定时间桶**：$T\_{due} = 3$，所属周 $W\_0 = \lfloor 3/7 \rfloor = 0$，双周滚动窗口为 $W\_0, W\_1$。
2.  **第一阶段硬扣减**：
    *   本周扣减：$Qty\_{take1} = \min(Demand.Qty, Allocation(Group\_A, W\_0).Available)$
    *   下周扣减：$Qty\_{take2} = \min(Demand.Qty - Qty\_{take1}, Allocation(Group\_A, W\_1).Available)$
    *   专有消纳量 $Allocated\_{private} = Qty\_{take1} + Qty\_{take2}$
3.  **第二阶段共享调剂**：
    *   若 $Demand.Qty - Allocated\_{private} > 0$，遍历所有其他组 $Group\_x \neq Group\_A$：
    *   调剂本周：$Qty\_{share1} = \min(Remaining, Allocation(Group\_x, W\_0).Leftover)$
    *   调剂下周：$Qty\_{share2} = \min(Remaining - Qty\_{share1}, Allocation(Group\_x, W\_1).Leftover)$
4.  **生成净需求**：
    *   $NetDemand = Demand.Qty - Allocated\_{private} - Allocated\_{shared}$

## 🔎 CASE 2：DOS 批量合并与多对一 Pegging 拆分 (DOS & M:1 Pegging)

### (1) 产品说明书（业务规则）
在物料需求计划中，为减少采购运输频率或生产换模次数，物料通常采用 **按天供应天数合并（Days of Supply, DOS）** 的批量规则。
*   **批量合并**：系统将 DOS 窗口内（如 5 天）产生的所有净需求合并为一笔计划订单（Planned Order），到货期落在窗口期的第一天。
*   **多对一无损追溯 (M:1 Pegging)**：虽然供应合并为一单，但系统在内存中必须保持**细粒度的需求原子追溯**，以便在计划控制塔中能够向下钻取，看清这笔采购单到底是为了满足哪几个客户的哪几笔订单。
*   **富余量结转**：批量整取（如向上取整到 Lot Size 100 的倍数）产生的富余库存，需结转至后续时序中被后续需求优先消耗。

### (2) 设计说明书（数据结构与计算步骤）
#### 核心数据结构：
*   `ipc_pegging_ledger` (供需 Pegging 关系表)
*   `ipc_planned_order` (计划订单表)

#### 算法求解步骤：
1.  **确定 DOS 窗口**：在 Day 1 发现首个净需求，DOS = 5，则合并窗口为 `[Day 1, Day 5]`。
2.  **汇聚需求量**：
    $$Qty\_{sum} = \sum\_{t=1}^{5} NetDemand(t) = 20 (Day 1) + 40 (Day 3) + 10 (Day 5) = 70$$
3.  **应用 Lot Size 限制**：物料 `Lot_Size = 100`。
    $$Qty\_{order} = \lceil 70 / 100 \rceil \times 100 = 100$$
4.  **生成 Planned Order**：
    *   到货日 = Day 1，数量 = 100。
5.  **构建内存 M:1 Pegging 链**：
    *   `PlannedOrder_01` (100) 分解分配给：
        *   `DEMAND_101` $\to$ 分配 20
        *   `DEMAND_102` $\to$ 分配 40
        *   `DEMAND_103` $\to$ 分配 10
        *   `Projected Excess` $\to$ 结转富余量 30
6.  **富余量时序结转**：
    *   在 Day 1 将富余量 30 写入 `Projected_On_Hand`。

### (3) 开发文档（输入/输出/对账 SQL）

#### 📥 1. 输入数据设置
*   物料：`PART_02`，标准提前期 = `0` 天，`OnHand = 0.0`
*   规则：`DOS = 5`，`Lot_Size = 100.0`
*   需求明细：
    *   `DEMAND_101`: Day 1, Qty = **20.0**
    *   `DEMAND_102`: Day 3, Qty = **40.0**
    *   `DEMAND_103`: Day 5, Qty = **10.0**

#### ⚙️ 2. C++ 内存求解步骤轨迹 (Trace Log)
```
[LBL DOS Netting Trace]
- Scanning Part: PART_02 (DOS: 5 days, LotSize: 100.0)
- Found first net demand on Day 1: 20.0
- Defining DOS Window: Day 1 to Day 5
  * Accumulating demand in window: Day 1 (20.0) + Day 3 (40.0) + Day 5 (10.0) = 70.0
- Applying Lot-sizing on accumulated 70.0 with LotSize 100.0:
  * Calculated Order Qty = ceil(70.0 / 100.0) * 100.0 = 100.0
- Generating Planned Order: PO_2001, Part: PART_02, Qty: 100.0, Due: Day 1
- Creating M:1 Pegging Records for PO_2001:
  * Segment 1: Qty 20.0 -> DEMAND_101 (Due Day 1)
  * Segment 2: Qty 40.0 -> DEMAND_102 (Due Day 3)
  * Segment 3: Qty 10.0 -> DEMAND_103 (Due Day 5)
  * Segment 4 (Excess): Qty 30.0 -> Projected On-Hand Day 1
```

## 🔎 CASE 3：ECN 时间卡尺过滤与新老料物理切换 (ECN Temporal Caliper)

### (1) 产品说明书（业务规则）
在离散制造中，产品结构经常发生工程变更（ECN）。例如：老芯片 `CHIP_V1` 只能用到 Day 10，之后必须强行切换为新芯片 `CHIP_V2`。
*   **静态拓扑无环**：为了在编译低层码（LLC）时避免时间维度的循环依赖，静态 BOM 关系将新老料全部编译，在静态图谱上不带时间属性。
*   **时空动态切换**：在具体物料需求消纳计算时，系统应用 **ECN 时间卡尺过滤**。对于需求日期为 $T\_D$ 的订单，只有当 $eff\_start\_day \le T\_D \le eff\_end\_day$ 时，对应的 BOM 边才生效，从而引导流量走向对应的新/老料。

### (2) 设计说明书（数据结构与计算步骤）
#### 核心数据结构：
*   `ipc_bom_item` (BOM条目表，带 `eff_start_day` 和 `eff_end_day` 属性)

#### 逻辑计算步骤：
1.  订单 `Order_A` 需求 $T\_{due} = 8$。
2.  下探 BOM，匹配子件 `CHIP_V1` (有效区间 `[0, 10]`)。由于 $0 \le 8 \le 10$，该边激活，传导需求。
3.  匹配子件 `CHIP_V2` (有效区间 `[11, 999]`)。由于 $8 < 11$，该边失效，剪枝。
4.  订单 `Order_B` 需求 $T\_{due} = 12$。
5.  匹配子件 `CHIP_V1` (有效区间 `[0, 10]`)。由于 $12 > 10$，失效，剪枝。
6.  匹配子件 `CHIP_V2` (有效区间 `[11, 999]`)。由于 $11 \le 12 \le 999$，激活，传导需求。

### (3) 开发文档（输入/输出/对账 SQL）

#### 📥 1. 输入数据设置
**BOM 配置（父项为 `PRODUCT_X`）：**
| parent_id | child_id | per_qty | eff_start_day | eff_end_day | 备注 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `PRODUCT_X`| `CHIP_V1`| 1.0 | `0` | `10` | 老芯片 |
| `PRODUCT_X`| `CHIP_V2`| 1.0 | `11` | `999`| 新芯片 |

**需求订单流：**
*   `DEMAND_301`: `PRODUCT_X`, Qty = **50.0**, due_day = **8**
*   `DEMAND_302`: `PRODUCT_X`, Qty = **80.0**, due_day = **12**

#### ⚙️ 2. C++ 内存求解步骤轨迹 (Trace Log)
```

- Processing parent demand PRODUCT_X on Day 8 (DEMAND_301, Qty 50.0):
  * Checking edge to CHIP_V1: Range [0, 10] -> Day 8 is IN. Edge ACTIVE.
    -> Generated dependent demand for CHIP_V1: Qty 50.0, due_day = 8.
  * Checking edge to CHIP_V2: Range [11, 999] -> Day 8 is OUT. Edge PRUNED.
- Processing parent demand PRODUCT_X on Day 12 (DEMAND_302, Qty 80.0):
  * Checking edge to CHIP_V1: Range [0, 10] -> Day 12 is OUT. Edge PRUNED.
```

## 🔎 CASE 4：多渠道采购份额分配与供应商独立节点层 (Option A Supplier Sourcing Quota Split)

### (1) 产品说明书（业务规则）
在供应链网络中，原材料通常有多个供应商，采购比例由合同配额决定（如供应商 A 占 60%，B 占 40%）。同时，供应商的生产产能也是受限的，并且具有特定的发运日历（发货时间窗口）。
*   **供应商独立节点（方案 A）**：系统在 BOM 最底层为每个 `[物料 + 站点 + 供应商]` 组合建立独立节点（如 `(RAW_01, SITE_01, SUPP_A)` 和 `(RAW_01, SITE_01, SUPP_B)`）。
*   **采购份额分流**：上游净需求到达后，复用 Method A 的配额纠偏算子按比例分流到供应商节点。
*   **供应商产能扣减**：分流后的采购量会检查该供应商对应的产能资源限制，若超载则进行时间平移或份额重分配。
*   **发运日历对齐**：计算计划采购单（Planned Purchase）的发运开工日时，必须根据供应商的特有日历向历史方向的有效发运日对齐，防止在供应商休息日下单。

### (2) 设计说明书（数据结构与计算步骤）
#### 核心数据结构：
*   `ipc_material_node` (物料节点，扩展 `supplier_code` 属性)
*   `ipc_supplier_calendar` (供应商发运日历表)
*   `ipc_supplier_capacity` (供应商产能表)

#### 逻辑计算步骤：
1.  原材料 `RAW_01` 在 Day 5 产生净缺口 100。
2.  **比例分摊**：
    *   供应商 A 分配：$100 \times 60\% = 60$
    *   供应商 B 分配：$100 \times 40\% = 40$
3.  **供应商 B 产能校验**：
    *   检查供应商 B 在 Day 5 的产能。已知最大供应上限为 30。
    *   超载裁决：将多余的 10 单位采购量重新调拨给供应商 A（供应商 A 采购量变更为 70）。
4.  **发运日历对齐**：
    *   供应商 A 的采购提前期 = 2天。发运日 $T\_{ship} = T\_{due} - LeadTime = 5 - 2 = 3$。
    *   检查供应商 A 的日历。Day 3 是周日（非发运日）。
    *   对齐：向历史方向寻找最近的有效发运日 Day 2（周六）。
    *   最终 Planned Purchase 订单：发运日 = Day 2，到货日 = Day 4。

### (3) 开发文档（输入/输出/对账 SQL）

#### 📥 1. 输入数据设置
**供应商节点与配额配置：**
| part_id | site_id | supplier_code | target_ratio | standard_lead_time |
| :--- | :--- | :--- | :--- | :--- |
| `RAW_01` | `SITE_01` | `SUPP_A` | **0.60** | 2 天 |
| `RAW_01` | `SITE_01` | `SUPP_B` | **0.40** | 2 天 |

**供应商产能与日历约束：**
*   `SUPP_B` 在 Day 5 的最大供应能力为 **30.0**。
*   `SUPP_A` 的发运日历：Day 3 为非发运日（无效），Day 2 为有效发运日。

#### ⚙️ 2. C++ 内存求解步骤轨迹 (Trace Log)
```
[LBL Supplier Sourcing Netting Trace]
- Net Demand for RAW_01 on Day 5: Qty 100.0
- Evaluating Sourcing Quota Split:
  * Target for SUPP_A: 100.0 * 0.60 = 60.0
  * Target for SUPP_B: 100.0 * 0.40 = 40.0
- Checking Supplier Capacity:
  * SUPP_B Capacity Check on Day 5: Max 30.0. Demand 40.0 exceeds limit by 10.0.
  * Clamping SUPP_B purchase to 30.0.
  * Reallocating overflow 10.0 to SUPP_A. SUPP_A demand becomes 70.0.
- Aligning Supplier Ship Calendars:
  * SUPP_B: LeadTime 2 days -> Ship Day = 5 - 2 = 3 (Valid ship day). PO_B ship = Day 3, due = Day 5.
  * SUPP_A: LeadTime 2 days -> Ship Day = 5 - 2 = 3. 
    -> Check SUPP_A Calendar: Day 3 is Sunday (Non-shipping).
    -> Sliding back to Day 2 (Saturday, Valid ship day).
    -> PO_A ship = Day 2, due = Day 4 (Earliest delivery).
```

## 🔎 CASE 5：虚拟物料在 LBL 中的零阻力直通爆开 (Phantom Parts Blow-Through)

### (1) 产品说明书（业务规则）
在复杂的离散制造中，为了规范 BOM 结构，通常会建立“虚拟物料（Phantom Parts）”（如：包装套件、CPU模块等）。
*   **物理本质**：虚拟物料本身不是实体，不进行实物入库与采购，其在手库存 OnHand 强制为 0.0，提前期 Lead Time 强制为 0.0。
*   **直通爆开**：当计划引擎扫描到虚拟物料时，直接跳过库存扣减与计划单生成，将净需求以“光速”穿透，按 BOM 配比与废品率 100% 传导到其底层的真实子组件上。

### (2) 设计说明书（数据结构与计算步骤）
#### 核心数据结构：
*   `ipc_material_node` (物料节点，设置 `is_phantom = true` 属性)

#### 逻辑计算步骤：
1.  父项 `FINISHED_GOOD` 在 Day 10 产生计划订单，需要虚拟件 `PHANTOM_KIT` 共 100 个。
2.  扫描 `PHANTOM_KIT`。由于 `is_phantom = true`：
    *   跳过 `OnHand` 消纳（即使账面有库存也不可用，强制按 0 计算）。
    *   提前期偏移量 $L\_{offset} = 0$。
3.  直接将 100 个需求穿透下钻给子件 `REAL_IC` (消耗系数 2.0，废损率 5%)。
    *   子需求数量：$100 \times 2.0 \times (1 + 0.05) = 210$。
    *   子需求日期：Day 10（时间轴无平移）。

### (3) 开发文档（输入/输出/对账 SQL）

#### 📥 1. 输入数据设置
**物料定义：**
| part_id | part_code | is_phantom | on_hand | standard_lead_time |
| :--- | :--- | :--- | :--- | :--- |
| `PART_FG` | `FINISHED_GOOD`| false | 0.0 | 2 天 |
| `PART_PH` | `PHANTOM_KIT` | **true** | **10.0** (强制不使用) | **0.0** |
| `PART_RL` | `REAL_IC` | false | 0.0 | 3 天 |

**BOM 装配关系：**
*   `FINISHED_GOOD` $\to$ `PHANTOM_KIT` (per_qty = 1.0, scrap = 0.0)
*   `PHANTOM_KIT` $\to$ `REAL_IC` (per_qty = 2.0, scrap = 0.05)

**毛需求：**
*   `PART_FG` 在 Day 12 需求 **100.0**。

#### ⚙️ 2. C++ 内存求解步骤轨迹 (Trace Log)
```
[LBL Phantom Blow-Through Trace]
- Net Demand for FINISHED_GOOD on Day 12: Qty 100.0
- Generating Planned Order for PART_FG: Qty 100.0, due = Day 12, start = Day 10 (LT: 2)
- Propagating demand from PART_FG to PHANTOM_KIT:
  * Requires PHANTOM_KIT on Day 10: Qty 100.0
- Processing PHANTOM_KIT (is_phantom = true):
  * Ignoring OnHand (10.0 is skipped). LeadTime = 0.
  * Direct blow-through to children of PHANTOM_KIT.
  * Target child: REAL_IC. 
    -> per_qty = 2.0, scrap = 0.05
    -> Calculated qty = 100.0 * 2.0 * (1 + 0.05) = 210.0
    -> Calculated due_day = 10 - 0 = Day 10
  * No Planned Order generated for PHANTOM_KIT.
```

| :--- | :--- | :--- | :--- | :--- | :--- |

## 🔎 CASE 6：多级库存安全库存优化 (MEIO) 的方差级联传播

### (1) 产品说明书（业务规则）
在多级供应链网络中，需求的不确定性（方差）会沿 BOM 有向无环图（DAG）逐级向下级联传播。如果只在成品层核算安全库存，中间件和原材料就会因缺乏库存缓冲而频繁断料。
*   **全网方差传播**：系统按照低层码（LLC）拓扑顺序自顶向下传播独立需求的均值与方差，将中间件及原材料的间接需求波动精确算透。
*   **服务水平锚定**：采用 Beasley-Springer-Moro 高精度有理逼近算法计算各物料的 Z-Score，并结合提前期波动计算各级安全库存。

### (2) 设计说明书（数据结构与计算步骤）
#### 核心数学公式：
*   **前向前向需求与方差级联公式**：
    对于父件 $P$ 到子件 $C$ 的 BOM 边，其有效转化乘数 $M = per\_qty \times (1 + scrap) \times target\_ratio$。
    $$D\_C += D\_P \times M$$
    $$\sigma^2\_{D,C} += \sigma^2\_{D,P} \times M^2$$
*   **安全库存公式**：
    $$SS\_i = \Phi^{-1}(SL\_i) \times \sqrt{ L\_i \cdot \sigma^2\_{D,i} + D\_i^2 \cdot \sigma^2\_{L,i} }$$

#### 逻辑计算步骤：
1.  成品 `PART_FG` 独立需求均值 $D\_{FG} = 100$，方差 $\sigma^2\_{D,FG} = 400$（标准差 20）。
2.  BOM 乘数 $M = 2.0$（废损率 0，配额比例 1.0）。
3.  下级中间件 `PART_SEMI` 的需求与方差传播：
    *   $D\_{SEMI} = 100 \times 2.0 = 200$
    *   $\sigma^2\_{D,SEMI} = 400 \times 2.0^2 = 1600$
4.  计算安全库存：
    *   中间件提前期 $L = 3$ 天，提前期方差 $\sigma^2\_L = 1.0$。
    *   服务水平 $SL = 0.95$，由 Beasley-Springer-Moro 近似计算出 $Z = 1.64485$。
    *   标准差整合项：
        $$\text{Combined\_StdDev} = \sqrt{3 \times 1600 + 200^2 \times 1.0} = \sqrt{4800 + 40000} = \sqrt{44800} \approx 211.66$$
    *   安全库存：
        $$SS\_{SEMI} = 1.64485 \times 211.66 \approx 348.15$$

### (3) 开发文档（输入/输出/对账 SQL）

#### 📥 1. 输入数据设置
**成品 `PART_FG` 独立需求特征：**
*   日均独立需求 $D\_{ind} = \mathbf{100.0}$
*   独立需求方差 $\sigma^2\_{D,ind} = \mathbf{400.0}$

**中间件 `PART_SEMI` 计划参数：**
*   服务水平目标 $SL = \mathbf{0.95}$
*   标准提前期 $L = \mathbf{3.0}$ 天，提前期方差 $\sigma^2\_L = \mathbf{1.0}$
*   BOM 装配比率 `per_qty = 2.0`, `scrap = 0.0`, `target_ratio = 1.0`

#### ⚙️ 2. C++ 内存求解步骤轨迹 (Trace Log)
```
[MEIO Calculation Trace]
- Starting MEIO forward propagation. Low Level Code 0 to Max.
- Processing Part: PART_FG (LLC 0)
  * Direct Demand: Mean = 100.0, Var = 400.0
- Propagating from PART_FG to PART_SEMI (LLC 1):
  * BOM Edge: Multiplier = 2.0 * (1 + 0.0) * 1.0 = 2.0
  * Propagating Mean: 100.0 * 2.0 = 200.0 -> PART_SEMI cumulative mean = 200.0
  * Propagating Variance: 400.0 * (2.0^2) = 1600.0 -> PART_SEMI cumulative var = 1600.0
- Calculating Safety Stock for PART_SEMI:
  * Service Level Target: 0.95 -> Beasley-Springer-Moro Z-score = 1.644853
  * Lead Time = 3.0 days, LT Variance = 1.0
  * Combined Uncertainty = sqrt(3.0 * 1600.0 + (200.0^2) * 1.0) = sqrt(4800.0 + 40000.0) = sqrt(44800.0) = 211.6601
  * Safety Stock (SS) = 1.644853 * 211.6601 = 348.15
- Writing to DuckDB.
```
